================================================================================
EVALUATION PLAN: Tessituragram-Based Repertoire Recommendation System
Research Questions for Offline Evaluation (No Human Data Required)
================================================================================

All three research questions use only: the song library (tessituragrams),
synthetic user profiles derived from that library, and the system's own
outputs.

--------------------------------------------------------------------------------
HOW THE THREE RESEARCH QUESTIONS RELATE
--------------------------------------------------------------------------------
The three RQs form a single evaluation story for your recommender:

  RQ1 — Does it work? When we "ask" for songs that match a given song's
        pitch profile, does the system put that same song at the top? (Accuracy.)

  RQ2 — Is it stable? When the user changes their preferences by one note,
        does the recommendation list stay mostly the same or completely
        reshuffle? (Robustness.)

  RQ3 — Do the numbers make sense? Do the scores spread out meaningfully,
        and do the score parts (similarity vs. avoid penalty) relate to
        the final score the way the formula says? (Interpretability.)

Same system, same data, same evaluation pipeline; three complementary
questions that together support the claim that the system is accurate,
stable, and interpretable.

--------------------------------------------------------------------------------
SYSTEM SUMMARY (components used in evaluation)
--------------------------------------------------------------------------------
- Data: data/tessituragrams.json. Each song has: composer, title, filename,
  tessituragram (MIDI note → duration in quarter beats), statistics
  (pitch_range.min_midi, pitch_range.max_midi). Source of MusicXML files:
  OpenScore Lieder Corpus (Gotham & Jonas, 2022; CC0, doi: 10.17613/1my2-dm23).
- Model: (1) Filter songs by user vocal range. (2) Build an "ideal" pitch
  vector from favourite and avoid notes (L2-normalised). (3) Score each
  song: cosine similarity between song's pitch distribution and ideal, minus
  a penalty for time spent on avoid notes: final_score = cos_sim − α×avoid_penalty.
  (4) Rank by final_score descending. Tie-break: filename (A–Z).
- Output: Ranked list with rank, filename, final_score, cosine_similarity,
  avoid_penalty, favorite_overlap per song.

================================================================================
RESEARCH QUESTION 1: Self-retrieval accuracy
================================================================================

1.1 Research question (specific and measurable)
   When we build a synthetic user profile from one song (using that song's
   most- and least-used pitches as favourites and avoids), does the system
   rank that same song at position 1, or within the top 3 or top 5?

1.2 What is being measured
   - For each song used as a "query," the rank position of that song in the
     system's recommendation list (1 = first, 2 = second, etc.).
   - Across all such queries: the fraction of times the query song is at
     rank 1 (Hit Rate @ 1), the average of 1/rank (Mean Reciprocal Rank),
     and the fraction of times it appears in the top 3 or top 5 (Hit Rate @ 3,
     Hit Rate @ 5).

1.3 Why it is being measured
   Your system recommends songs by similarity of pitch distribution to an
   ideal. If the ideal comes from one song, that song is the only correct
   "match" — no human judgment needed. Measuring how often it appears at
   the top tests whether the scoring and ranking are internally consistent.
   This is standard offline evaluation for content-based recommenders when
   you do not have human labels (Herlocker et al., 2004; Urbano et al., 2013).

1.4 How it will be measured

   Model: The same code as your app: filter_by_range, build_ideal_vector,
   score_songs from src/recommend.py. Use α = 0.5; tie-break by filename.

   Data: data/tessituragrams.json (101 songs). Format: JSON array of song
   objects with tessituragram and statistics.pitch_range. Define the **valid
   query pool** as every song in the library for which, after filtering by
   that song's range, at least 2 songs remain (so there is a real ranking).
   For the experiment, draw up to N_QUERIES songs at random from this pool
   (with a fixed random seed for reproducibility).

   Procedure:
   1. Load the library from JSON.
   2. For each randomly sampled song s in the valid query pool: set user
      range to s's min/max MIDI. From s's tessituragram (L1-normalised),
      take the top 4 MIDI by duration as favourites and the bottom 2 as
      avoids (if fewer than 4 pitches, use all as favourites; if fewer than
      2, use no avoids). The sampled pool is drawn from songs whose filtered
      candidate set has at least 2 songs.
      In the paper, state explicitly: the synthetic profile is derived from
      the same L1-normalised pitch-duration representation used when
      scoring songs (internal coherence).
   3. Run the recommender; get ranked list. Find the rank of s (by filename).
      The query song s must remain in the candidate set (do not remove it).
      Using s's own range for filtering ensures s is in the candidate set.
   4. Record: hit@1 (1 if rank = 1, else 0), 1/rank, hit@3, hit@5.
   5. After all sampled songs: HR@1 = mean(hit@1), MRR = mean(1/rank),
      HR@3 = mean(hit@3), HR@5 = mean(hit@5).

   Evaluation: Report HR@1, MRR, HR@3, HR@5. Report 95% confidence intervals
   (bootstrap over the query set, e.g. 10,000 samples). If the library is
   small (< 30 valid queries), say so as a limitation.

   Formulas:
   - HR@1 = (1/N) × (number of queries where the query song was rank 1).
   - MRR = (1/N) × sum over queries of (1 / rank of query song).
   - HR@k = (1/N) × (number of queries where the query song was in top k).

1.5 Plain-language summary (limited math/CS background)
   Think of it like this: you pick a song and tell the system "recommend
   songs that match this one's vocal profile." The most sensible top answer
   is that same song. For the experiment, the computer pretends that a
   **random sample** of songs from the library are different users' favourite
   songs (drawn from all songs where the system can produce a real ranking).
   For each of these sampled songs, we see where it appears in its own
   recommendation list. We then count: how often did that song come first?
   How often in the top 3 or top 5? "Hit Rate @ 1" is the fraction of times
   it came first; "Mean Reciprocal Rank" rewards putting it as high as
   possible (first is best). We also report a range of plausible values
   (confidence interval) so readers know how stable the result is. No people
   are asked to judge anything—only the stored pitch data and the program
   are used.

RESEARCH QUESTION 2: Ranking stability under small preference changes
================================================================================

2.1 Research question (specific and measurable)
   When we change the user's favourite or avoid list by exactly one note
   (add one or remove one), how similar is the new recommendation list to
   the original? We measure similarity with Kendall's τ (tau): 1 = identical
   order, 0 = no relationship, −1 = completely reversed.

2.2 What is being measured
   - The similarity between two rankings of the same set of songs: the
     original ranking and the ranking after one note is added or removed
     from favourites or avoids.
   - We repeat this for many one-note changes and report the average and
   spread of τ (and a 95% confidence interval for the baseline-level mean τ).

2.3 Why it is being measured
   In practice, users might add or forget one note. A stable system should
   not completely reorder the list for such a tiny change—that supports
   trust and interpretability. This is a standard robustness check;
   evaluation literature stresses stability of results under small changes
   (Herlocker et al., 2004; Urbano et al., 2013).

2.4 How it will be measured

   Model: Same as RQ1 (filter_by_range, build_ideal_vector, score_songs;
   α = 0.5; tie-break by filename).

   Data: data/tessituragrams.json (101 songs). Use up to 5 baseline user
   profiles (N_BASELINES = 5), each derived from a different song using the
   same rule as RQ1 (top-4 fav, bottom-2 avoid, disjoint). Baselines are
   sampled at random (with a fixed seed) from all songs whose filtered
   candidate set has at least 10 songs (MIN_CANDIDATES = 10). Using multiple
   baselines reduces the risk that results depend on an atypical baseline.

   Procedure:
   1. For each baseline, compute the baseline ranking R0 over the candidate
      set C (all songs that pass filter_by_range for that profile).
   2. For each one-note perturbation on that baseline:
        - add one favourite (choose a MIDI note that lies within the user's
          range, occurs in at least one song in C, and is not already in the
          favourite or avoid lists),
        - or remove one favourite,
        - or add one avoid (same MIDI constraints as add-favourite),
        - or remove one avoid.
      Keep the same vocal range so C does not change; favourites and avoids
      are enforced to remain disjoint in all perturbations.
   3. For each perturbation, compute the new ranking R_new over the same C.
   4. For each pair (R0, R_new), compute Kendall's τ (e.g. scipy.stats.kendalltau
      on the two rank vectors for the same set of songs).
   5. Compute:
        - the mean and standard deviation of τ across all perturbations
          (descriptive), and
        - the mean τ per baseline, plus a 95% confidence interval for the
          baseline-level mean (bootstrap over baseline means, resampling
          baselines with replacement).

   Evaluation: Report mean τ and 95% CI. Interpret: τ > 0.7 strong agreement;
   0.3–0.7 moderate; < 0.3 weak. You can state a hypothesis (e.g. mean τ ≥ 0.5)
   and check whether the CI supports it.

   Formula: Kendall's τ = (C − D) / (n(n−1)/2), where C = number of song
   pairs ordered the same way in both lists, D = number ordered differently,
   n = number of songs. τ is between −1 and 1 (Kendall, 1948).

2.5 Plain-language summary (limited math/CS background)
   We ask: "If the singer changes their list of favourite or avoid notes by
   just one note, does the recommendation list stay roughly the same or
   get completely shuffled?" We compare the original list order with the
   new list order using a number τ (tau): 1 means the order is the same,
   0 means there's no clear relationship, and negative would mean the order
   flipped. We try many such one-note changes and report the average τ. A
   high average means the system is stable—small input changes do not
   cause big output changes.

RESEARCH QUESTION 3: Score spread and formula checks
================================================================================

3.1 Research question (specific and measurable)
   (a) Do recommendation scores (final_score) spread out across songs, or do
   many songs get almost the same score? (b) Does the scoring formula behave
   as specified (final_score = cos_sim − α×avoid_penalty)? We separate
   formula/identity checks from descriptive sanity-check correlations.

3.2 What is being measured
   (a) Spread: variance and range of final_score over the ranked list for
       each run (computed on unrounded scores); we report the average
       variance and range across M runs.
   (b) Formula checks (strong): (i) Identity residual: final − (cos − α×avoid)
       per song; report mean and max absolute residual over runs. (ii) Optional
       per-run regression final ~ cos + avoid; report mean coefficients (expect
       ≈ 1, ≈ −α) and mean R² (expect ≈ 1).
   (c) Sanity-check correlations (descriptive): Pearson r for final_score vs
       cosine_similarity and final_score vs avoid_penalty; Spearman rho for
       cosine_similarity vs favorite_overlap. Correlations are aggregated
       via Fisher z (see below). Runs with undefined correlation (e.g. zero
       variance in one variable) are excluded for that pair; n_excluded is
       reported. Expected signs: final–cos positive; final–avoid negative;
       cos–favorite_overlap positive.

3.3 Why it is being measured
   If scores barely differ, the ranking is not very discriminative. If the
   formula does not hold numerically (identity residual or regression), the
   implementation may not match the design. The sanity-check correlations
   confirm that the pipeline and rounding have not introduced anomalies.
   Evaluation literature emphasises discriminative power and construct
   alignment (Herlocker et al., 2004; Urbano et al., 2013; Castells et al., 2018).

3.4 How it will be measured

   Model: Same scoring logic as the app (build_dense_vector, normalize_l1,
   build_ideal_vector, cosine_similarity) but computed in the experiment
   without rounding, so variance/range/correlations use raw values.

   Data: data/tessituragrams.json (101 songs). M = 25 synthetic user
   profiles (N_PROFILES = 25), each derived from a different song with the
   same rule as RQ1 (top-4 fav, bottom-2 avoid, disjoint). Profiles are
   sampled at random (fixed seed) from all songs whose filtered candidate
   set has at least 10 songs (MIN_CANDIDATES = 10).

   Procedure:
   1. Build all valid profiles; randomly sample M (with seed).
   2. For each profile: compute scores (unrounded) for each song in the
      candidate set: final_score, cosine_similarity, avoid_penalty,
      favorite_overlap.
   3. Per run: (a) Variance and range of final_score. (b) Identity residual
      = |final − (cos − α×avoid)|; mean and max. (c) If n ≥ 3 and both cos
      and avoid have positive variance: regression final ~ cos + avoid;
      record intercept, coef_cos, coef_avoid, R². (d) Pearson r for
      final_score vs cosine_similarity and final_score vs avoid_penalty;
      Spearman rho for cosine_similarity vs favorite_overlap. Return NaN
      when correlation is undefined (e.g. zero std).
   4. Across runs: mean variance, mean range, mean identity residual, max
      residual; mean regression coefficients and mean R² (over runs with
      valid regression). For correlations: aggregate via Fisher z (r
      clamped to (−1, 1) before atanh; average z, optionally weighted by
      n_songs − 3; back-transform with tanh); 95% CI by bootstrap over
      runs (resample runs, then Fisher-aggregate each bootstrap sample).
   5. Report n_excluded per correlation (runs where that correlation was NaN).

   Bootstrap: The unit of analysis is the run (profile). We bootstrap by
   resampling the list of runs with replacement. A hierarchical bootstrap
   (resample profiles then songs within profile) could be used in future
   work to reflect song-level variability if reviewers request it.

   Evaluation: Report spread (mean variance, mean range, 95% CI); identity
   check (mean and max residual); regression summary (mean coefs, mean R²);
   sanity-check correlations (Fisher-mean and 95% CI, n_excluded). Frame
   final↔component correlations as pipeline sanity checks, not as strong
   "internal validity" evidence; the identity and regression checks are the
   primary formula checks.

   Formulas:
   - Variance = (1/(n−1)) × sum over songs of (score − mean_score)².
   - Range = max(score) − min(score) over the list.
   - Pearson r, Spearman ρ: standard definitions (−1 to 1).
   - Fisher z: z = atanh(r) with r clamped to (−1, 1); mean z (optionally
     weighted by n−3); aggregate r = tanh(mean z). See e.g. Fisher (1915),
     or meta-analysis references in SOURCES below.

3.5 Plain-language summary (limited math/CS background)
   We check two things. First: do the scores the system gives to different
   songs actually differ? We measure how spread out the scores are
   (variance and range). Second: does the scoring formula behave as
   intended? We verify that the final score really equals "similarity minus
   penalty" (identity check) and optionally run a small regression to
   confirm the weights. We also report correlations between the parts of
   the score as a sanity check (e.g. higher similarity goes with higher
   scores). We use many synthetic users (randomly chosen), aggregate
   correlations in a statistically sound way (Fisher z), and report
   confidence intervals. No human input is used—only the numbers the
   system computes.

--------------------------------------------------------------------------------
NOTE ON BOOTSTRAPPING (applies to all three RQs)
--------------------------------------------------------------------------------
All three experiments report "95% bootstrap confidence intervals." Here is
what that means in plain language:

Imagine you ran the experiment and got a result — say, "the system ranked
the correct song first 85% of the time." But if you had a slightly different
set of songs, or different synthetic profiles, the number might come out as
83% or 87%. How much could it reasonably vary?

Bootstrapping answers this by simulating "what if we ran the experiment many
times with slightly different data?" It works like this:

  1. You have your original set of results (e.g. 101 rank values, one per
     song).
  2. You randomly draw 101 values from that set *with replacement* — meaning
     the same result can be picked more than once, and some results may be
     left out. This simulates getting a slightly different dataset.
  3. You compute the metric (e.g. HR@1) on this resampled set.
  4. You repeat steps 2–3 ten thousand times (10,000 resamples).
  5. You now have 10,000 slightly different versions of the metric. You sort
     them and take the 2.5th percentile as the lower bound and the 97.5th
     percentile as the upper bound. This is your 95% confidence interval.

The interval tells you: "we are 95% confident the true value falls somewhere
in this range." A narrow interval means the result is stable and reliable; a
wide interval means there is more uncertainty.

Why bootstrapping instead of a formula? Traditional formulas for confidence
intervals assume the data follows a specific mathematical shape (like a bell
curve). Bootstrapping makes no such assumption — it lets the data speak for
itself. This makes it a good default choice, especially for metrics like
hit rates and rank-based scores that do not follow a bell curve.

All three experiments in this repository use the same bootstrap procedure
(10,000 resamples, random seed 42 for reproducibility).

--------------------------------------------------------------------------------
LIMITATIONS (state these in your paper)
--------------------------------------------------------------------------------
- **No human judgements.** Evaluation is fully offline. Synthetic user
  profiles are derived from the song library; there is no ground-truth
  relevance from real users. Results indicate internal consistency and
  robustness of the scoring pipeline, not user satisfaction.

- **Single library and corpus.** The experiment uses one dataset
  (data/tessituragrams.json, 101 songs from the OpenScore Lieder Corpus).
  Findings may not generalise to other repertoires, larger libraries, or
  different musical styles.

- **RQ1.** The valid query pool is all songs with at least 2 candidates
  after range filtering. If this pool is small (e.g. fewer than 30 songs),
  report it as a limitation: the bootstrap confidence intervals will be
  wider and the estimates less precise. The experiment samples up to
  N_QUERIES (e.g. 50) from this pool at random.

- **RQ2.** Only a small number of baselines (e.g. 5) are used;
  perturbations are restricted to one-note changes. The 95% CI is for the
  baseline-level mean τ (bootstrap over baselines), not over individual
  perturbations.

- **RQ3.** The bootstrap is at the run (profile) level, not hierarchical
  (e.g. resampling profiles then songs within profile). Correlations
  between final_score and its components (cosine_similarity, avoid_penalty)
  are by design sanity checks that the pipeline behaves as implemented,
  not evidence of "internal validity" in the strong sense; the identity
  check and regression (final ≈ cos − α×avoid) are the primary formula
  checks. If any runs are excluded due to undefined correlations (e.g.
  zero variance), n_excluded is reported.
